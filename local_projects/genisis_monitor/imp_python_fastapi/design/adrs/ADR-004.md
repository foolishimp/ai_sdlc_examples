# ADR-004: Full-Page Navigation for Design Tenant Switching + HTMX configRequest for Filter Propagation

**Status**: Accepted
**Date**: 2026-03-03
**Feature**: REQ-F-MTEN-001
**Drives**: REQ-F-MTEN-002, REQ-F-MTEN-003

---

## Context

The project detail dashboard shows data from a shared event log that may contain events from multiple design tenants (e.g. `imp_claude`, `imp_gemini`, `imp_codex`). The backend already supports `?design=` filtering on all 14 fragment routes via `_get_historical_state()`. Two UX decisions are required:

1. **How should design tenant switching work?** Options: HTMX swap vs full page navigation.
2. **How should the active design filter propagate to all fragment requests?** Options: modify each `hx-get` URL individually vs a single global interceptor.

---

## Decision 1: Full-Page Navigation for Tenant Switching

Design tenant tabs are plain `<a href="...">` links that perform a full page navigation, not HTMX swaps.

### Alternatives Considered

1. **HTMX swap** — swap the entire `#main-content` div via `hx-get`. Would preserve the SSE connection, but requires the selector to know which elements to swap, and risks stale per-fragment `hx-get` URLs that don't yet carry the new `?design=` param.
2. **Full page reload** (chosen) — the selector renders with `?design=<tenant>` in the href. On click, the browser reloads the full project page with the correct filter. All fragments re-render consistently. The page script reinitialises `_currentDesign` from the new URL.

**Chosen**: Full page navigation. Design switching is a significant context switch — the user is moving between independent implementations. A full reload is semantically correct and avoids the fragment-ordering hazard where some sections render before the design param propagates.

---

## Decision 2: HTMX `configRequest` Interceptor for Filter Propagation

A single `htmx:configRequest` event listener injects the `design` parameter into every HTMX request when a tenant is active, rather than modifying each of the 14 `hx-get` attributes.

```javascript
document.addEventListener('htmx:configRequest', function(evt) {
    const design = new URLSearchParams(window.location.search).get('design');
    if (design) {
        evt.detail.parameters['design'] = design;
    }
});
```

### Alternatives Considered

1. **Modify each `hx-get` URL individually** — e.g., generate `hx-get="/fragments/project/X/graph?design={{ design }}"` for all 14 fragment sections. Correct but brittle: any new fragment added in future must remember to include the param. High maintenance surface.
2. **Single `configRequest` interceptor** (chosen) — fires for every HTMX request including SSE-triggered reloads. Adding new fragment routes in future requires no change. Clean separation: fragment `hx-get` attributes stay minimal; the interceptor handles cross-cutting param injection.

**Chosen**: `configRequest` interceptor. Single point of truth for design filter propagation.

---

## Decision 3: Scrubber `triggerReload()` Explicit Design Param

The scrubber's `triggerReload()` function uses `htmx.ajax()` directly (not declarative HTMX attributes), so the `configRequest` interceptor does not fire for scrubber-triggered reloads. The scrubber must build the query string explicitly:

```javascript
const _currentDesign = new URLSearchParams(window.location.search).get('design') || '';

const triggerReload = () => {
    const parts = [];
    if (!isLive) parts.push("t=" + encodeURIComponent(window.eventTimestamps[eIdx]));
    if (_currentDesign) parts.push("design=" + encodeURIComponent(_currentDesign));
    const tParam = parts.length ? "?" + parts.join("&") : "";
    htmx.ajax("GET", basePath + tParam, {target: target});
};
```

`_currentDesign` is captured once at page load. Because tenant switching is a full page reload (Decision 1), the captured value is always current.

---

## Consequences

**Positive**:
- Zero per-fragment `hx-get` changes — all 14 fragment routes work automatically
- Adding new fragment routes in future requires no design-filter boilerplate
- Full-page tenant switch ensures consistent fragment state across all sections
- Pico CSS `aria-current="page"` provides active tab styling natively — no custom CSS class needed

**Negative**:
- Full page reload on tenant switch interrupts the SSE connection momentarily (reconnects within ~1s via SSE built-in reconnect)
- `configRequest` does not fire for programmatic `htmx.ajax()` calls — scrubber must handle design param explicitly
